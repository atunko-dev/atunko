## Context

`RecipeInfo` is currently a Java record that extracts individual fields (name, displayName, description, tags, recipeList) from OpenRewrite's `RecipeDescriptor` during discovery. `RecipeDescriptor` has a rich `options` field (`List<OptionDescriptor>`) that is silently dropped.

`RunConfig` stores recipes as `List<String>` (names only). `RecipeExecutionEngine.execute(String, List<SourceFile>)` activates recipes by name with default options.

OpenRewrite 8.74.1 provides `Recipe.withOptions(Map<String, Object>)` which clones a recipe and applies option values via Jackson deserialization — this is the official programmatic API for setting options at runtime.

## Goals / Non-Goals

**Goals:**
- Expose option metadata on `RecipeInfo` by wrapping `RecipeDescriptor`
- Store per-recipe option values in `RunConfig`/`.atunko.yml`
- Apply option values during execution via `Recipe.withOptions()`
- Display option metadata in TUI and Web UI detail panels
- Provide a JSON Schema for `.atunko.yml`

**Non-Goals:**
- Interactive option editing in TUI (issue #17 — depends on this work)
- Option validation beyond what OpenRewrite provides
- Favorites, recently used, or custom recipe YAML (other #10 sub-features)
- Backward compatibility with version-1 `.atunko.yml` files (pre-1.0 project)

## Decisions

### 1. RecipeInfo wraps RecipeDescriptor (not extends, not copies fields)

**Choice**: `RecipeInfo` becomes a record with components `(RecipeDescriptor descriptor, List<RecipeInfo> recipeList)` and provides delegation methods (`name()`, `displayName()`, `options()`, etc.).

**Why over field extraction**: Forward-compatible — any new `RecipeDescriptor` field is accessible without changing `RecipeInfo`. Eliminates the redundant data model. Options (and future fields like `examples`, `maintainers`) come for free.

**Why not just use RecipeDescriptor directly**: `RecipeDescriptor.getRecipeList()` returns `List<RecipeDescriptor>`, but we need recursive wrapping to `List<RecipeInfo>`. Also, `RecipeInfo` provides a test-friendly `of()` factory that avoids `RecipeDescriptor`'s 14-argument Lombok constructor.

**Why record not class**: Immutability is correct here — recipe metadata doesn't change. The record gives us `equals`/`hashCode` for free.

### 2. Static factory `RecipeInfo.of()` for test ergonomics

**Choice**: `RecipeInfo.of(name, displayName, description, tags)` and `RecipeInfo.of(name, displayName, description, tags, recipeList)` construct a `RecipeDescriptor` with sensible defaults internally.

**Why**: Tests currently use `new RecipeInfo("name", "display", "desc", Set.of("tag"))`. Without a factory, they'd need to construct 14-field `RecipeDescriptor` objects. The factory preserves test readability.

### 3. New RecipeConfig record for RunConfig

**Choice**: `RecipeConfig(String name, Map<String, Object> options)` replaces `String` in `RunConfig.recipes`.

**Why not a map at RunConfig level**: Per-recipe options are naturally co-located with the recipe name. A `Map<String, Map<String, Object>>` would separate names from their options and complicate ordering.

**YAML format**:
```yaml
version: 1
recipes:
  - name: org.openrewrite.java.ChangeMethodName
    options:
      newMethodName: baz
  - name: org.openrewrite.java.SimplifyBooleanExpression
```

### 4. Keep RunConfig version at 1

**Choice**: Don't bump the version number despite the schema change.

**Why**: The project is pre-1.0.0 (unstable). Versioning the internal config format adds ceremony with no consumer to protect. `@JsonIgnoreProperties(ignoreUnknown = true)` already provides forward tolerance.

### 5. Overloaded execute method (not replacing existing)

**Choice**: Add `execute(String recipeName, Map<String, Object> options, List<SourceFile> sources)` alongside the existing `execute(String, List<SourceFile>)`.

**Why**: The no-options path remains clean for callers that don't need options. The existing method delegates to the new one with an empty map.

### 6. Recipe.withOptions() for applying values

**Choice**: Use OpenRewrite's `Recipe.withOptions(Map<String, Object>)` — activates by name first, then applies options.

**Why not YAML injection**: `withOptions()` is the official API, uses Jackson internally for type coercion, and handles edge cases (case-insensitive properties, list splitting). Building synthetic YAML declarations would be fragile and duplicate logic OpenRewrite already provides.

## Risks / Trade-offs

- **RecipeDescriptor constructor stability** → The 14-field `@AllArgsConstructor` constructor is generated by Lombok and could change across OpenRewrite versions. Mitigation: the factory is isolated in `RecipeInfo.of()` — a single point to update. Production code uses `new RecipeInfo(descriptor, recipeList)` which only depends on the wrapper.

- **`Recipe.withOptions()` clones via Jackson** → If a recipe has non-serializable state, `withOptions()` could fail. Mitigation: this is OpenRewrite's own API; if it fails for a recipe, that's an upstream bug. We can catch and surface the error gracefully.

- **Breaking `.atunko.yml` format** → Existing config files (if any) won't load. Mitigation: acceptable pre-1.0. The JSON Schema will be the format reference going forward.
